# 浏览器插件二期实施方案

## 文档目标

这份文档用于定义浏览器插件二期的实施范围、模块拆分、阶段目标与验收标准。

二期不再解决“一期能不能跑通”的问题，而是解决以下 4 个核心问题：

1. 插件如何从本地优先模式演进为远端优先、云端可调度的产品形态
2. 插件如何把外部 AI 调用、自身内置对话和未来桌面端协同收拢到同一套运行体系
3. 插件如何把临时引导会话沉淀成可复用的任务模板与流程资产
4. 插件如何具备可交付、可运营、可审计的产品化能力

## 二期总体目标

二期目标不是继续堆单点功能，而是把浏览器插件升级成 `Immersive-Input` 体系中的浏览器执行端与浏览器入口端。

二期完成后，应具备以下能力：

- 插件支持 `remote / local` 双连接模式，默认 `remote`
- 外部 ChatGPT / Claude / MCP 与插件内置聊天都能统一进入同一套任务运行时
- 云端可以管理设备、路由任务、追踪会话、分发模板
- 浏览器端可以保存、加载、回放、版本化 Guide 模板
- 浏览器插件与未来桌面端 `Immersive-Input` 能进行真实 handoff / resume
- 产品具备登录、权限、审计、日志和异常恢复能力

## 二期范围

### 纳入二期

- 远端连接模式与本地模式并存
- 云端网关、设备绑定与会话路由
- 内置聊天 provider 抽象与云端模型接入
- Guide 模板化、版本化、追溯与回放
- 浏览器到桌面端协同与 handoff
- 产品化基础能力：登录、权限、审计、状态页

### 不纳入二期

- 桌面端完整自动化能力的全面实现
- 多租户企业后台的完整管理系统
- 面向所有第三方网站的通用模板市场
- 完整商业化计费系统

## 二期架构目标

二期建议采用“入口层 + 云端控制层 + 本地执行层 + 跨端桥接层”的结构。

### 1. 入口层

- 外部 AI 入口：ChatGPT / Claude / MCP
- 内置对话入口：插件 popup / sidepanel / companion
- 未来桌面端入口：`Immersive-Input`

### 2. 云端控制层

- 账号与鉴权
- 设备注册与设备在线状态
- 任务分发与会话路由
- Guide 模板服务
- 会话追踪、事件审计与回放索引

### 3. 本地执行层

- 浏览器插件 Guide Runtime
- Overlay 渲染与交互
- 页面选择器解析与目标定位
- 浏览器动作执行与状态回传

### 4. 跨端桥接层

- 浏览器到桌面端 handoff
- 桌面端接管后的 resume
- 统一会话 ID、任务 ID、桥接事件历史

## 二期模块拆分

### A. 连接模式模块

目标：让插件同时支持本地与远端，但默认走远端。

建议新增：

- `app/chrome-extension/common/connection-types.ts`
- `app/chrome-extension/common/connection-mode.ts`
- `app/chrome-extension/entrypoints/background/connection-manager/`
- `app/chrome-extension/entrypoints/popup/components/connection/`
- `app/chrome-extension/entrypoints/sidepanel/components/connection/`

核心能力：

- 保存当前连接模式
- 默认远端连接
- 远端地址、设备 ID、登录态与连接状态展示
- 本地 bridge 作为高级模式保留

### B. 云端网关模块

目标：让云端成为任务控制面，插件成为浏览器执行面。

建议新增：

- 云端 `gateway` 服务
- 设备 `registry` 服务
- 任务 `session-router` 服务
- 浏览器插件远端 `transport`

核心能力：

- 设备上线、下线、心跳
- 会话绑定到具体浏览器设备
- 外部 AI 请求被路由到设备
- 浏览器端结果与事件回传云端

### C. AI 入口统一模块

目标：统一外部 AI 调用和内置聊天入口。

建议新增：

- `provider` 适配层
- `cloud-model` provider
- `user-api-key` provider
- `external-mcp` provider

核心能力：

- 内置对话可调用用户 API Key 或云端托管模型
- 外部 MCP 与内置聊天共享同一任务载荷协议
- 保留任务来源、模型来源和上下文来源

### D. 模板系统模块

目标：让 Guide 从“临时会话”变成“可复用资产”。

建议新增：

- `guide-template` 领域模型
- 模板存储、版本、发布状态
- 模板与锚点的浏览器侧编辑入口
- 模板回放与历史版本回看

核心能力：

- 从 Session 保存为模板
- 从模板创建新 Session
- 模板版本化
- 模板适用范围、页面约束、锚点约束
- 模板执行历史与失败记录

### E. 浏览器到桌面协同模块

目标：让浏览器插件与 `Immersive-Input` 形成一套混合执行链路。

建议扩展：

- `guide-runtime/bridge-adapter.ts`
- 共享桥接事件协议
- handoff 触发条件与 handoff UI
- desktop resume 元数据

核心能力：

- 浏览器执行不了的步骤可转交桌面端
- 桌面端完成后可恢复到浏览器 Session
- 事件历史不中断
- 用户能看到当前步骤由哪个执行端接管

### F. 产品化基础能力模块

目标：让插件具备上线、演示、交付和后续运营能力。

建议新增：

- 登录与设备绑定页
- 连接状态页
- 审计与会话时间线页
- 失败重试与错误提示层
- 权限确认与敏感操作确认

核心能力：

- 用户知道当前是本地还是远端模式
- 用户知道当前设备是否在线
- 用户知道当前会话是谁发起、在哪执行、执行到哪一步
- 用户可以看到失败原因、重试与恢复建议

## 二期实施阶段

### 阶段 2.1：连接模式与远端优先

目标：

- 插件支持 `remote / local`
- 默认 `remote`
- UI 可见连接状态

交付：

- 连接模式存储
- `ConnectionManager`
- popup / sidepanel 连接入口
- welcome 页切换为远端优先说明

验收：

- 新安装插件默认进入远端引导
- 用户可以切换到本地模式
- 插件能正确显示当前连接状态

### 阶段 2.2：云端网关与设备路由

目标：

- 云端可识别浏览器设备
- 外部 AI 任务可被路由到在线设备

交付：

- 设备注册
- 设备心跳
- 任务路由协议
- 浏览器端远端 transport

验收：

- 云端能看到在线设备
- 任务可投递到指定浏览器设备
- 浏览器执行结果可回传云端

### 阶段 2.3：模板化与回放

目标：

- Guide Session 可沉淀为模板
- 模板可复用、版本化、可回放

交付：

- 模板保存/加载
- 模板版本管理
- 模板执行历史
- 模板转 Session 与 Session 转模板

验收：

- 同一任务能基于模板重复执行
- 用户可查看模板版本
- 用户可查看历史执行记录

### 阶段 2.4：浏览器与桌面端协同

目标：

- 浏览器与 `Immersive-Input` 可跨端接力

交付：

- handoff UI
- bridge 状态页
- resume 能力
- 浏览器 / 桌面执行角色标识

验收：

- 浏览器可发起 handoff
- 桌面端可接管并回写状态
- 会话不中断

### 阶段 2.5：产品化交付能力

目标：

- 插件具备发布与持续运营的基础能力

交付：

- 登录/绑定
- 审计与时间线
- 权限确认
- 错误恢复提示

验收：

- 用户能够理解当前连接模式、设备状态与执行状态
- 敏感操作有明确确认
- 失败时有可见的恢复路径

## 二期验收标准

二期完成时，应满足以下标准：

### 功能标准

- 外部 AI 与内置聊天均可创建任务并进入统一运行时
- 插件默认支持远端模式，且可切回本地模式
- Guide 模板可保存、复用、版本化
- 浏览器端与桌面端可进行 handoff / resume

### 工程标准

- 二期新增协议继续沉淀到共享层
- 云端与本地 transport 有明确边界
- 浏览器执行层不直接绑定某个单一模型平台
- 构建、编译与关键链路验证可持续执行

### 产品标准

- 用户能理解“当前连接到哪里、由谁执行、做到哪一步”
- 用户能切换模式、查看状态、理解错误
- 系统具备审计、追溯与最小恢复能力

## 二期风险与把关点

### 风险 1：远端化后职责边界混乱

把关点：

- 云端只做控制面
- 浏览器插件保留本地执行权
- 不让云端直接替代浏览器动作执行层

### 风险 2：模板系统过早复杂化

把关点：

- 二期模板先围绕高频任务
- 不在二期做开放式模板市场
- 先支持保存、版本、回放与追溯

### 风险 3：浏览器与桌面端桥接耦合过深

把关点：

- 所有跨端状态都走共享协议
- 不在浏览器层写死桌面端细节
- 保持 handoff / resume 的协议边界

### 风险 4：产品入口过多导致 UI 再次发散

把关点：

- 所有入口统一汇聚到 Guide Runtime
- popup、sidepanel、welcome 页只做入口与状态展示
- 后台能力不直接散落到多个 UI 页面各自调用

## 二期启动前置条件

- 一期文档状态切换为已收口
- 当前 `compile` 与 `build` 继续保持通过
- 当前共享协议冻结一版作为二期起点
- 浏览器插件当前入口与运行时结构不再回退到“各入口各自实现”
