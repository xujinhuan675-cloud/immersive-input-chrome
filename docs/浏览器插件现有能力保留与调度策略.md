# 浏览器插件现有能力保留与调度策略

## 目的

这份文档回答一个关键问题：

当前项目里已经有很多实用能力，一期做沉浸式引导时，到底哪些要删除，哪些要保留，哪些应该转到后台作为可调度能力。

结论先说：

- 一期不建议硬删除成熟能力
- 建议做“前台收口，后台保留”
- 主线 UI 只保留和沉浸式引导强相关的入口
- 其余成熟能力尽量保留为后台可调度工具，供 AI、Guide Runtime、未来 `Immersive-Input` 调用

## 处理原则

### 原则 1：不轻易删成熟能力

现有很多能力虽然不适合作为一期主界面中心，但它们本身非常适合作为执行层基础设施，例如截图、读页面、注入脚本、点击、填值、流程执行、触发器、网络捕获等。

这些能力一旦删除，后面再接入 AI 或桌面端时还得重新补一遍，成本会更高。

### 原则 2：前台收口，后台保留

一期应该让用户感知到的核心只有：

- 任务入口
- 当前引导进度
- 页面中的高亮与说明
- 必要的权限确认

其余复杂能力不一定要在前台大量暴露，但应该保留在后台作为可调度能力。

### 原则 3：能做基础设施的能力优先保留

越接近以下角色的能力，越不应该删除：

- 页面理解
- 元素定位
- 用户交互执行
- 运行时调度
- 调试与审计
- 未来桌面端桥接

## 能力分类结论

## 一类：必须保留，且适合后台调度

这些能力建议继续保留，并逐步纳入 Guide Runtime / AI Runtime 的可调度能力层。

### 浏览器执行工具

对应代码：

- `app/chrome-extension/entrypoints/background/tools/browser/`

建议保留的能力：

- 页面导航
- 读页面 / 可访问性树
- 页面截图
- 点击、输入、键盘事件
- 注入脚本与脚本通信
- JavaScript 执行
- 控件选择与元素拾取
- 文件上传
- 对话框处理
- 下载处理
- 网络请求与网络捕获
- 多标签页 / 窗口控制

原因：

- 这些是浏览器执行层最核心的工具箱
- Guide Runtime、MCP、未来桌面桥接都会反复复用这些能力
- 它们非常适合作为“AI 可调度动作”

### Record-Replay 执行与调度能力

对应代码：

- `app/chrome-extension/entrypoints/background/record-replay/`
- `app/chrome-extension/entrypoints/background/record-replay-v3/`

建议保留的能力：

- Flow 执行
- Run 日志
- Queue / Scheduler
- Trigger 管理
- 重试策略
- 运行时恢复
- 运行结果回放

原因：

- 这些已经很接近任务编排内核
- 一期不一定把完整 Builder 暴露给用户，但后台调度价值非常高
- Guide Session 未来可以直接复用其中的调度与审计思想

### Native Host / 本地桥接

对应代码：

- `app/native-server/`
- `app/chrome-extension/entrypoints/background/native-host.ts`

建议保留原因：

- 这是浏览器插件和本地能力之间的桥梁
- 未来浏览器插件和 `Immersive-Input` 的衔接，很可能会继续依赖这层

## 二类：保留，但前台入口要收口

这些能力不建议删除，但不应该继续让主界面过于分散。

### AgentChat

对应位置：

- `sidepanel/components/AgentChat.vue`

建议：

- 保留
- 继续作为内置 AI 入口
- 后续接入 `Guide Session Runtime`

原因：

- 它是你“双入口”架构里“自有入口”的核心
- 不能删，但表达方式应该从“独立聊天功能”升级为“任务发起与进度反馈入口”

### Element Marker 管理

对应位置：

- `entrypoints/background/element-marker/`
- `sidepanel` 中的标注管理 UI

建议：

- 保留存储和验证能力
- 保留 UI，但地位下降
- 后续升级为 `Guide Anchor` 管理能力

原因：

- 它本质上就是后续 Guide 锚点资产
- 这部分非常值钱，不应该丢

### Web Editor

对应位置：

- `entrypoints/background/web-editor/`
- `web-editor-v2`

建议：

- 保留底层覆盖、定位、选区、变更跟踪能力
- 一期不把“完整页面编辑器”作为主叙事

原因：

- 沉浸式引导需要覆盖层、选区、高亮和局部说明
- 这些能力和 Guide Overlay 高度相关

### Quick Panel

对应位置：

- `entrypoints/background/quick-panel/`

建议：

- 保留
- 作为轻触发入口和上下文助手
- 不作为一期主入口中心

原因：

- 它很适合承接“从页面局部直接发起任务”
- 后续可以变成“从元素发起 Guide”的快捷入口

## 三类：保留为辅助能力，但不进入一期主线

这些能力建议先别删，但暂不放到一期主流程。

### 本地语义模型 / 向量检索

对应位置：

- `semantic-similarity`
- `vector-search`
- `popup` 本地模型页

建议：

- 保留
- 作为内容理解、页面检索和知识增强能力
- 不作为一期主闭环依赖

原因：

- 有价值，但不是“一期 Guide 跑起来”的阻塞项

### GIF、性能分析、书签、历史记录

对应位置：

- `gif-recorder`
- `performance`
- `bookmark`
- `history`

建议：

- 保留为附加工具
- 不作为一期前台主入口

原因：

- 这些能力对调试、分析、演示、辅助任务有用
- 但不是沉浸式引导的一期主线能力

## 一期前台建议保留的入口

用户可见层建议先收口到 4 类：

1. 内置 AI 任务入口
2. Guide Preview / Guide Session 入口
3. 任务进度与步骤状态
4. 锚点管理入口

不建议一期继续把以下能力放在主界面中心：

- 完整 Workflow Builder
- 大而全的工具菜单
- 与 Guide 主线弱相关的模型管理入口

这些能力可以保留，但应弱化露出。

## 一期后台建议继续可调度的能力

下面这些建议保留成后台能力池，未来可由 AI / Guide Runtime / `Immersive-Input` 统一调用：

- `chrome_navigate`
- `chrome_screenshot`
- `chrome_get_web_content`
- `chrome_get_interactive_elements`
- `chrome_click_element`
- `chrome_fill_or_select`
- `chrome_keyboard`
- `chrome_inject_script`
- `chrome_send_command_to_inject_script`
- `chrome_javascript`
- `chrome_read_page`
- `chrome_request_element_selection`
- `chrome_network_capture`
- `chrome_network_request`
- `chrome_upload_file`
- `chrome_handle_dialog`
- `chrome_handle_download`
- `record_replay_flow_run`
- `record_replay_list_published`

## 当前建议

如果按你的真实目标推进，我建议：

- 不删除原项目里的成熟执行能力
- 一期只收口前台入口和主线叙事
- 把浏览器插件做成“Guide Runtime + Tool Runtime”
- 把原有成熟能力沉到底层，供 AI 和未来桌面端调度

一句话总结：

当前项目里很多你觉得“其实还需要”的功能，确实不该删；它们更适合从“前台单独功能”转为“后台统一可调度能力”。
